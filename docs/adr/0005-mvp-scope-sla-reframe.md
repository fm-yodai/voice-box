# ADR-05: MVP スコープ確定と「放置しない」担保メカニズムのリフレーム

**ステータス:** 暫定

## コンテキスト

inception deck は思想・「やらないこと」・優先順位（品質 > スコープ > 期日）を定義していたが、**最初にリリースする最小機能セット（positive scope）と Done の定義が無く**、Sprint に落とせない状態だった。"MVP" を冠する Issue は #6（最小アーキ）/ #9（開発基盤）でプロダクト機能の Epic が存在しなかった。

加えて inception deck Q2/Q3/Q7/Q9 は **「一次受付 SLA」を差別化の柱**として暗黙に位置づけていたが、オーナー判断で「SLA はそこまで重要ではない（人力では努力する）」と再評価された。2026-05-15 のオーナーヒアリングで以下を確定。

## 検討した選択肢

| 選択肢（スコープ深さ） | メリット | デメリット |
|------------------------|---------|-----------|
| 投稿＋公開画面の閲覧のみ | 最小 | 「対応が見える」核の価値が出ない（却下） |
| 投稿＋公開画面＋運営ステータス更新（採用ベース） | 透明性の核が成立する最小 | — |
| ＋ SLA 自動計測・可視化 | 放置抑止が明示的 | SLA を重視しない方針と不整合・実装増（却下） |
| ＋ AI 対話的意見構築（採用） | 経営層可読性と投稿者の負担軽減を両立する差別化 | グリーンフィールドで実装比重大 |

## 決定

1. **MVP positive scope = Epic A〜D**: 匿名投稿＋利用者識別 / AI 対話的意見構築 / 公開画面 / 運営ステータス管理。受け入れ基準と実装ギャップは [docs/mvp-scope.md](../mvp-scope.md) に定義
2. **利用者識別**: ログイン必須・`authorId` を常時紐付け（公開経路に出さない）。`PostSchema` で構造担保済みのため新規実装不要
3. **AI 介在範囲を確定**（inception deck Q4 の「あとで決める」を解消）: 「投稿時の対話的意見構築」に限定。短文起点 → AI がドラフト提示 → 投稿者がフィードバック再生成または直接編集 → **投稿者の確定操作で初めて提出**。投稿者の確定なしの自動提出はしない（Q4「全自動はしない」と整合）。声の保全は「投稿者の確定権」で構造担保
4. **「その声、放置しない。」の担保メカニズムをリフレーム**: `SLA 計測` → `透明性（公開画面での投稿・対応ステータス可視化）`。SLA の自動計測・期限表示・超過通知は MVP 非対象。SLA は `docs/operations.md` の人力 **努力目標**（旧 §2「サービスレベル合意」→「対応目標」に改称）として残す
5. **SLA 計測は非重視**（Non-Goal）。キャッチコピーと価値提案は維持し、担保の柱を透明性に一本化

## 旧判断の変更

本 ADR は inception deck Q2/Q3/Q7/Q9 の「一次受付 SLA を差別化の柱とする」という暗黙の決定を変更する。当該判断は ADR ではなく思想文書だったため、新規 ADR 追加（本 ADR）＋思想・紹介文書の整合修正で対応した:

- 整合修正済み: `README.md` / `CLAUDE.md` / `docs/operations.md` / `inception-deck.md`（Q2/Q3/Q4/Q5/Q7/Q9）
- `.claude/skills/project-knowledge/SKILL.md` の本体 L15・description L5 はエージェントの self-modification ガードでブロックされたため、**ユーザーが手動修正して整合済み**（L7/L9 の「SLA」は検索トリガー語として意図的に残置）

## 結果

- 良い帰結: Sprint に落とせる受け入れ基準が揃った。スコープを品質優先で絞り込めた。匿名性担保が既存スキーマの追認で済むことが判明
- 注意すべき帰結:
  - SKILL.md 不整合の解消はユーザー対応待ち
  - 「経営層可読性 vs 投稿者の声の保全」は投稿者の確定権で担保するが UX 検証が必要
  - AI 対話は実装痕跡ゼロのグリーンフィールドで MVP の重心。技術選定（LLM 連携）は別途 Spike/ADR を要する可能性
  - ステータスは暫定。実装で妥当性確認後 `採用` に更新
